PRD 작성을 위한 토론 규칙

웹앱 기획 전문가 역할로, 초보자에게 맞춰 전문용어는 해설을 붙이고, 한 번에 하나씩만 질문한다. 답변은 객관식 선택 또는 단답형으로 받는다.

용어 해설
PRD (Product Requirements Document)
제품 요구사항 문서. 무엇을 왜 만드는지 정리한 기획서.
FS (Functional Specification)
기능 명세서. PRD의 내용을 화면·데이터·동작 단위로 쪼갠 개발용 설계서.
진행 규칙 확정 — “됐어”라고 할 때까지 검토와 검증을 이어간다.

질문 07 · 데이터베이스 선택
Claude

Firebase 데이터베이스 종류는 무엇을 사용하나요?

FS 문서 작성 전 마지막 기술 확인 항목입니다.

용어 해설
Firestore
데이터를 폴더(컬렉션)와 문서 구조로 저장. 검색·필터링에 강합니다.
Realtime Database
데이터를 하나의 큰 JSON 트리로 저장. 실시간 동기화에 특화되어 있습니다.
  1. Firestore (추천)
  2. Realtime Database
  3. 어떤 걸 써야 할지 모르겠음, 추천대로 진행
답변
1 — Firestore
품목 필터링(대분류 · 중분류 · 소분류)이 핵심 기능이므로 Firestore가 적합.
현재 진행 중 · 미확정 항목
Claude

신청 취소 시 데이터를 어떻게 처리할까요?

FS 문서 작성 완료. 남은 결정 사항 중 첫 번째입니다.

  1. 삭제 — 기록을 남기지 않음, 구조가 단순해짐
  2. “취소됨” 상태로 보존 — 이력 관리, PRD의 영구보관 원칙과 일치
대기 중
아직 답변하지 않은 질문입니다.
남은 미확정 항목 3가지
  • 신청 취소 시 삭제 여부 (현재 질문)
  • 자동 상태전환 · 자동 독촉 스케줄러 구현 방식
  • 엑셀 업로드 오류 데이터 처리 방식
